## Tarea 1

1. Limitaciones de hardware al instalar modelos grandes
- No se pudo instalar el modelo llama3 o mistral completo debido a que la memoria y potencia de la computadora no alcanzaban.
- Se utilizó un modelo más "lite": llama3.2:1b.

2. Parámetros malformados en herramientas
- El modelo en ocasiones envía un JSON con propiedades mal estructuradas, por ejemplo:
  { "final_answer": "..." } en lugar de { "text": "..." }.
- Esto genera errores como:
      final_answer() got an unexpected keyword argument 'final_answer'
- Solución: mapear automáticamente "final_answer" -> "text" en la función use_tool.
  (Es decir, corregir el nombre de la clave para que coincida con el parámetro de la función.)

3. Uso incorrecto de final_answer en lugar de code_exec
- Ejemplo: “calcula promedio” devolvió solo:
      Calculating the monthly sales average (y) from the given dataset.
  No hizo el cálculo, solo explicó.
- Causa: el modelo no eligió la herramienta correcta.
- Solución: reforzar el SYSTEM_PROMPT para indicar que primero debe usar code_exec y luego final_answer.

4. Problemas con nombres de variables y columnas
- El modelo generaba código con "df" en lugar de "dtf".
- También aparecían errores por mayúsculas/minúsculas en nombres de columnas (Age vs age, Score vs score).
- Solución: implementar un sanitizador de código que:
  - Reemplace "df" -> "dtf".
  - Convierta mean(), max(), min(), sum() en métodos de pandas.
  - Normalice nombres de columnas a los correctos de dtf.


5. Respuestas fuera del DataFrame
- Cuando la consulta no está relacionada con dtf, el modelo intentaba usar code_exec.
- Solución: ajustar el prompt para que:
  - Use search_web si requiere información externa.
  - Use final_answer si es una definición o explicación general.

6. Limitación de idioma
- El sistema funciona principalmente en inglés.
- En español puede dar errores de interpretación o elegir mal las herramientas.
- Solución: usar inglés para preguntas críticas y reforzar el prompt si se quiere compatibilidad con español.
